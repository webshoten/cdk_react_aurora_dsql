# 15. Design

## 設計フェーズの目的
- 目的を達成するための実装方式を決める。
- どの処理をどの層・モジュールが持つかを決める（責務境界）。
- どのディレクトリ・ファイルへ置くかを決める（配置方針）。
- インフラ構成をどうするかを決める。
- フロントエンドの画面構成（画面分割・導線・状態）を決める。

## 設計ルール
- 合意と不確実性の扱い
  - 設計で確定していない推測実装はしない。
  - ユーザーとの合意が未完了の事項は、提案までに留めて実装へ進めない（提案→合意→実装）。
  - 実施状況や確度は事実ベースで明示し、不確実な内容を断定しない。
- 責務分離と配置方針
  - まず `docs/overview/13.coding-architecture/overview.md` を参照して、責務境界と配置の前提を確認する。
  - 責務分離（Stack/Construct、domain/layer など）は、実装前に境界と配置方針を明示して合意する。
  - 配置判断に迷う場合は、編集前に責務境界と配置先を文章で確定してから実装する。
  - ファイル配置は「責務の所属先ディレクトリへ閉じる」を原則とし、同一責務を親子ディレクトリへ分散配置しない。
  - `index.ts` は公開面（export / 接続）に限定し、実装本体は個別モジュールへ配置する。
- CDK 設計
  - CDK は「その場しのぎ」の回避コードを禁止し、恒久的な設計で解決する。
  - AWS CLI 等でリソース状態を直接更新して問題解決しない。恒久対応は CDK 定義へ反映して deploy で揃える。
  - CDK で扱う構成値・依存値の解決は、手動 CLI 前提の運用で補わず、再 deploy で同じ結果に収束する冪等構成で設計する。
  - CDK は Stack 本体や `constructs/*/index.ts` にロジックを集中させず、責務ごとに Construct / module を分離する。
  - CDK の `constructs/*/index.ts` / `constructs/**/index.ts` は compose 専用とする。
  - CDK の `constructs` 配下は「利用機能名」ではなく「作成・参照する AWS リソース責務」単位で分割する。
  - 各リソースディレクトリの `index.ts` は入口（compose）とし、具体処理は同ディレクトリ内の別モジュールへ分離する。
  - 複数リソースをまたぐ共通化は、単独リソース責務を崩さない範囲に限定し、まずは各リソースディレクトリ内で閉じる。
- CDK 変更の事前確認
  - CDK 変更に着手する前に、必ず「今、deploy/destroy は実行中ですか？」を確認する。
  - deploy/destroy 実行中は CDK 変更を停止して待機する。
  - stateful リソース（DB/Bucket など）をリファクタする場合は、`cdk diff` で置換有無を確認し、置換が出る変更は事前合意なしで実施しない。
- DSQL 設計
  - Aurora DSQL 向け migration の DDL は、実装前に一次情報（AWS 公式）で対応可否を確認する。
  - `ALTER TABLE ADD COLUMN` の制約同時追加（`DEFAULT` / `NOT NULL`）非対応を前提に、列追加と制約付与を分けて設計する。
  - `SERIAL` 非対応のため、連番カラムは `IDENTITY` 前提で設計する。
  - 同一 transaction で DDL と DML を混在できない前提で、DDL 実行と履歴更新 INSERT を分離設計する。
- 設計フェーズの進行
  - 作業は「設計タスク化」と「実装タスク化」を分離して進める。
  - 設計フェーズは `設計タスク化 → 設計更新 → 設計読み合わせ合意` を完了してから実装へ進む。
  - 設計タスク化では、各タスクに完了条件（どの設計書に何を反映し、何を合意できれば完了か）を記載する。

## ドキュメント記述ルール（overview）
- 記載対象
  - `overview.md` は「これから作る予定」ではなく「現時点の確定状態」を書く。
  - `overview.md` は「あるべき姿（確定仕様）」のみを記載し、実装タスクは記載しない。
  - 未確定事項は `overview.md` に書かず、`plan/` または `7.open-issues` で管理する。
- 記述形式
  - セクションは「要約1行 + 子リスト（具体）」で記述する。
  - 役割説明は `<要素名>: <役割の要約>` で統一する。
  - 子リストには「実際に何をしているか」を 2〜4 点で記載する。
- 記述内容
  - 抽象語だけで終わらせず、判定条件・入出力・抑制項目などの事実を書く。
  - 利用者目線で記述し、「どこで何を確認・操作すればよいか」を読んで即判断できる表現にする。
  - 対象実装の前提を知らない人でも読めるように、状態名・関数名だけで終わらせず、いつ何が起きるか・なぜその挙動にするかを短く説明する。
  - 構成を書くときは「配置する/追加する」のような予定表現を使わず、要素名とファイルパスを記載する。
- 禁止事項
  - `TODO` / `YYYY-MM-DD` のプレースホルダを残さない。
  - 実装手順・移行手順・作業分解などのタスクは `plan/` 配下の別 `*.md` に分離する。
  - 経緯を記載する場合は本文へ混在させず、該当項目の子リストに「補足」として簡潔に記載する。

## 図表管理（drawio）
- 正本と配置
  - 図の正本は `.drawio` とする。
  - 配布・参照用は `.svg` を同一ディレクトリに配置する。
- 更新手順（Docker 前提）
  - `.drawio` を更新したら `.svg` を再生成し、両方を同一タスクで更新する。
  - 変換は Docker を利用して実行する（ローカルの draw.io desktop CLI 依存にしない）。
  - 例: `docker run --rm -v "$PWD:/data" -w /data rlespinasse/drawio-export <path/to/file.drawio> --format svg`
- 運用ルール
  - 図を `md` / `tsx` で参照する場合は `.svg` を参照する。
  - `.drawio` のみ変更して `.svg` を更新しない状態を残さない。
