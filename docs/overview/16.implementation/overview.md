# 16. Implementation

## 実装フェーズの目的
- 設計で合意した内容を、迷いなく実装へ落とし込む。
- 実装で問題があれば、いつでも設計フェーズへ戻って見直す。
- 設計の前提自体に問題がある場合は、要件フェーズへ戻って見直す。

## 実装ルール
- 実装着手前の確認
  - 実装着手前に `docs/overview/13.coding-architecture/overview.md` を参照し、責務境界と配置方針の前提を確認する。
- 既存流用と重複防止
  - 既存実装で流用できるところは流用する。
  - 同じ責務の処理を新規に重複実装しない。
  - 新規実装が必要な場合は、先に共通化可能性を確認する。
  - 流用しづらい共通メソッドは、相談の上で共通化可否を確認してから修正する。
- スコープ管理
  - `YAGNI` を優先し、要求されていない機能や返却値を足さない。
  - `YAGNI` は機能追加の抑制に使い、設計品質は本番運用前提（可観測性・保守性・安全性）で担保する。
- 実装時の品質
  - exported 関数・クラスには責務が分かる短いコメントを付与し、処理意図が読み取れる状態を維持する。
  - React コードを実装・レビューする場合は、`react-best-practices` Skill の指針を参照して判断する。
  - DB実装は、原則 `drizzle-orm` を利用する（例外は理由をコメントまたは設計に残す）。
- ドキュメント反映
  - 実装完了時は、合意した内容を関連ドキュメントへ反映する。
  - 変更に伴う構成・出力契約・手順の更新は同一タスクで反映する。
